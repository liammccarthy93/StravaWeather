Analysing The External Influences on Strava Performances¶

Task 1: Choosing an API¶

Strava is an application on which cyclists track their cycles. One of its main attractions is "segments". These are user-created sections of road on which times are recorded by the application and leaderboards are created. Positioning on some of these segments can be highly competitive and 1st place rankings (known as King of the Mountains, or KOMs, regardless of whether the segment is actually a hill or not) are highly sought after. While fitness and cycling ability are important factors in segment performance, there are also a range of external factors which influence the leaderboard, such as wind, or using the aerodynamic effects of other riders. This project shall investigate the influence of these factors on segment performance and aims to conclude on whether segment position is actually about cycling ability, or merely who had the most favourable conditions.

The Strava API provides the data underlying the Strava website, and will be used extensively in this project. There are many available APIs for weather, however many of the historical data is actually the forecast for that time, and not actual measured data. Ireland's weather service, Met Eireann, provides hourly measured data from a range of weather stations. In this project, the data from Dublin Airport will be used, as it includes a lot of different measurements. Met Eireann do not have an API, but the data is freely available to download from their website in CSV form.

Task 3: Data Storage¶

A data frame is initialised to store the data for each effort. It will be filled with the data in 'efforts'. Since 'efforts' is an generator object, it is not possible to tell how many rows the data frame must be. The row corresponding to each effort will have to be appended one at a time. This is not ideal in terms of efficiency, as many appends means there is a large amount of reallocation of memory required, however it is neccessary in this case due to the unknown size of the data.

Each Effort object in the iterator can then be iterated through and the relevant object attributes are written to the data frame. Please note that due to the large size of the data and the relatively inefficient method which must be used to create the data frame, this may take a minute or two. This data frame is then written to a CSV file to allow for access later.

The full CSV of weather data is 250,000 rows. Only a small subset of these rows are of interest, namely those that correspond to the segment efforts above. Therefore it would be a poor use of memory resources to load in the entire dataset as a data frame. A database which could be efficently queried would be much more suitable. PyTables provides a powerful and effective database system for use with Python.

Firstly, a class must be defined with the column type for each column of the database.

In [8]:
class weatherData(tables.IsDescription):
	date	    = tables.StringCol(50)   # The date and time of recording in string form
	wdspInd		= tables.UInt8Col()      # integer between 1 and 6, which represents whether the recording is reliable
	wdsp  		= tables.UInt16Col()     # the wind speed
	wddirInd	= tables.UInt8Col()      # indicator for direction measurement, with same scale as above
	wddir  		= tables.UInt16Col()     # the wind direction

The database file is then created. The BLOSC compression library is used as this library achieves excellent read/write speeds. The PyTables documentation suggests that read/write speeds may be faster with compression than without. The minimum compression level is used because disk space is not a limitation.

In [9]:
# open DB file for writting - creates if it doesn't already exist, overwrites if it does.

# create a table to store the weather data
table = h5filePtr.create_table("/", "DublinWeather", weatherData, "DublinWeather", filters=filters)

Task 4: Data Pre-Processing¶

Clean data¶

The segment data can now be read from the corresponding CSV file. Since the CSV file saves everything as text, the datetime and timedelta columns must be manually converted to that value. This allows the data to be sorted by elapsed time, creating a leaderboard. Thus, the index of the effort is the position on the leaderboard.

We only want to consider genuine efforts, and not efforts in which the athlete was not trying hard, or where a mechanical issue hindered them a lot. Thus, a cutoff time of twice as slow as the fastest time was set as the threshold. Any efforts over this threshold are considered to not be full efforts and thus are discarded.

The cutoff time is not an ideal way to measure whether a segment effort was a full effort or not, however other measures such as power meter output or heart rate monitoring are not available for many efforts, and they vary hugely for different athletes with different ages and physiologies.

The following code loads the data from the CSV file and applies the above cleaning measures to it.

# function creates an array that repeats the numbers 0-23 for a provided number of days (default 7)
def hoursOfTheDay(days=7):
    return np.array([[y for y in range(24)] for x in range(days)]).flatten()

# function that creats an array of the numbers 0-6 that repeat 24 times each, in a row, for a given number of days
def daysOfTheWeek(days=7):
    return np.array([[x for y in range(24)] for x in range(days)]).flatten()

# function to group a dataset which has hour and day of week columns into the number of efforts at those times
def createHourAndDayGroups(original_df):
    # create a copy to preserve integrity of data frame that was passed in.
    original_Copy = original_df.copy()
    
    for column in original_Copy.columns:
        # remove column if it doesn't match any of the below column names
        if column != 'StartTime' and column != 'Day' and column != 'Hour':   
            original_Copy = original_Copy.drop(column, axis=1)

    original_Copy_groups = original_Copy.groupby(['Day', 'Hour']) # Group by day and hour
    original_Copy_groups = original_Copy_groups.count()           # sum up the number of enteries for StartTime.
                                                                  # This gets number of efforts for that hour
    # replace StartTime column label with number of efforts
    original_Copy_groups.columns = [col.replace('StartTime', 'NumEfforts') for col in original_Copy_groups.columns]
    
    index = [daysOfTheWeek(), hoursOfTheDay()]                   # create list of the correct indexes
    original_Copy_groups = original_Copy_groups.reindex(index)   # reindex using these indexes
    original_Copy_groups = original_Copy_groups.fillna(0)        # Rows where there were no efforts are NaN
                                                                 # this is replaced with 0.
    return original_Copy_groups

Weather¶

Weather data can be added to the efforts data frame. Weather data for each hour is contained in the weather database, and can easily be accessed using SELECT style commands. Weather records are added for the hour of each segment effort in this manner.

Weather data may be unreliable in some cases. This is indicated by a 2 in either of the Indicator fields. Weather data is not written to the data frame for these cases, and so the values remain as NaN. The NaN values can then be removed from the weather data frame, as there is no corresponding reliable data available.

Prepare data for comparing group and solo efforts¶

The data in the efforts_df data frame will now be organised to analyse how the prescence of others on the segment influences segment performance.

Firstly, the data will be grouped by minute of the day. This will roughly group efforts together. The limitation of this method is that it will seperate efforts made together but which occur either side of a change in minute. However, this was decided to be an exceptional case, and the trade-off of losing this data was acceptable for the efficient method of grouping.

These groups were then split into two seperate data frames, one which contains solo efforts, where that effort had no one else start that minute and finish within the cut-off time. Group efforts are where more than 1 person started that minute and finished within the cut-off time.

# function that takes a dataframe of efforts and creates the values to be plotted in a heatmap
def heatmapColours(df):
    hours = range(24)       # hours of the day
    dayNames = range(7)      # days of the week
    
    # set x and y to show data for each time slot. Showing N values requires N+1 positions in the mesh-grid.
    x, y = np.meshgrid(np.concatenate((dayNames, [dayNames[-1]+1])), np.concatenate((hours, [hours[-1]+1])))
    # initialise colours array with 0 for each position.
    colours = np.array([[0 for w in range(len(dayNames))] for z in hours])
    #coloursSH = np.reshape(df.values,(len(hours), len(dayNames))).swapaxes(0,1)

    # turn the two value for each column into a 2D array
    i,j,= 0,0              # initialise counters
    for day in dayNames:   # for every day
        for hour in hours: # for each hour in that day
            # transfer the value in the DF into the 2D array
            colours[j,i] = df.loc[(day, hour), 'NumEfforts']
            j+=1
        i+=1
        j=0

    return x,y,colours

Conclusions¶

The results above show that the fastest times on this Strava leaderboard are dominated by efforts which benefited from highly favourable wind conditions. It was found that solo cyclists performed better than group cyclists. This is not because being alone is faster than being in a group, but rather the results suggest that the increased flexibility of riding alone allows solo cyclists to better capitalise on opportune weather conditons, thus more than making up for the lost aerodynamic benefits while cycling alone.

Based on the results of this project, the advise to any cyclists aiming to record an exceptional time on this segment would be to wait for a favourable wind, and to make the most of it while its optimal rather than waiting from help from a group.
